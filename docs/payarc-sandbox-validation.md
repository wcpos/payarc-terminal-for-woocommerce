## Mock contract references

The docs-derived fixture currently models these shapes:

- `sale_request`: `tenantId`, `terminalId`, `transactionId`, `tenderType`, amount object, `printReceipt`, `callbackURL`, and metadata.
- `sale_response_success_initiated`: synchronous `traceId` plus response status. This synchronous `traceId` behavior is not live-verified.
- `get_transaction_success`: authoritative transaction shape with `transactionId`, `status`, `chargeId`, amount object, card details, processor details, timestamp, `traceId`, and metadata.
- `callback_success`, `callback_decline`, `callback_timeout`: callback-shaped transaction payloads. The plugin must treat callbacks as triggers and verify successful completion through get-transaction before marking an order paid.
- `cancel_request`, `cancel_response_success`, `cancel_response_failure_already_processed`: cancel-by-`traceId` shapes. Cancel behavior is not live-verified.

## Secure setup requirements

Use real PayArc sandbox credentials only through secure WordPress admin configuration or another approved secret channel. Never commit bearer tokens, callback secrets, real tenant IDs, or real terminal IDs.

Required settings for validation:

1. PayArc test API bearer token.
2. PayArc callback bearer token expected in callback `Authorization` headers.
3. PayArc tenant ID, expected by the current local validator to be exactly 12 digits.
4. PAX terminal ID, expected by the current local validator to be exactly 10 digits.
5. Public HTTPS callback URL generated by the plugin and reachable by PayArc.

The gateway admin diagnostics table displays non-secret values only: mode, Connect base URL, masked tenant ID, masked default terminal ID, webhook URL, last callback timestamp, and last PayArc error code/message after token-like text redaction.

The **Validate Settings** admin control performs local checks only. It does not call PayArc. It checks that tokens are configured, tenant and terminal IDs match local formats, the callback URL uses HTTPS, and receipt/tender enum values are valid.

## Sandbox validation sequence

1. Activate **PayArc Terminal for WooCommerce**.
2. Enter PayArc test bearer token, callback bearer token, tenant ID, and PAX terminal ID in WooCommerce payment gateway settings.
3. Confirm the callback URL is public HTTPS and reachable by PayArc.
4. Start a test order payment from POS or WooCommerce order-pay.
5. Confirm whether PayArc returned `traceId` synchronously or only via callback, and verify the plugin follows the Task 0 decision:
   - If synchronous `traceId` is returned, polling and cancel may use it.
   - If no synchronous `traceId` is returned, the plugin must wait for callback/get evidence and must not invent a trace ID.
6. Complete an approved transaction on the terminal.
7. Confirm the callback leads to authoritative get-transaction verification, marks the order paid, and stores `chargeId`.
8. Run decline, timeout, duplicate callback, and cancel-before-card-present scenarios.
